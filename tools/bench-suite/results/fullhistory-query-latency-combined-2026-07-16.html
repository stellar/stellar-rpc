At a glance

On the light endpoints, query adds only ~4–18 ms on top of ingest. A direct hot read of a 17 MB ledger (incl. zstd decode) is ~8 ms, yet serving it over HTTP is ~567 ms — ~98.6 % is base64 + JSON + HTTP shipping the blob, ~1.4 % is the storage read. 02

How to read this report

Every latency below is milliseconds (ms), lower is better, measured end-to-end over JSON-RPC unless a row says otherwise. Definitions of every term and column header:

Hot tier vs cold tier
hot = the daemon's live RocksDB store, holding recent ledgers in the "hot" DB. cold = frozen, immutable chunk artifacts for older history — a zstd .pack file plus MPHF + roaring-bitmap index files. A query hits one tier depending on how old the target ledger is.
p50 / p90 / p99 / max
Latency percentiles (nearest-rank) over the request sample. p50 = median (half the requests were faster); p90 / p99 = the slowest 10 % / 1 % boundary; max = the single slowest request. p50 tells you the typical case, p99 the tail.
count / reps
Number of requests in the sample. Light endpoints use 500 reps; getLedgers is capped by a 45-second wall budget (so its count is however many requests fit — 71–127), which makes its p99 less robust.
Endpoints
getLedgers = fetch whole ledger(s) including the full LedgerCloseMeta blob (large — 10–17 MB each here, measured at limit=1). getTransaction = one transaction by hash. getTransactions = a page of transactions. getEvents = a filtered contract-event query.
Ingest (source → durable commit)
Time from the daemon receiving a ledger to it being durably committed (one fsync). This is the "how fresh" leg — separate from query latency, which reads already-committed data.
conc / req/s (Part B)
conc = number of concurrent in-flight requests the load generator keeps open. req/s = completed requests per second (throughput) at that concurrency.
term / OR-union (getEvents sweep)
A term is one indexed constraint — a distinct contract ID, or a distinct topic value at a position. A query OR-unions N terms (matches any of them). window = the query spans the 20-ledger steady-state window; chunk = random starts over the whole 10,000-ledger frozen chunk.

05

Combined analysis

getLedgers is payload-bound everywhere. Live pubnet (~1.7 MB LCM) → ~35 ms end-to-end; synthetic (~17 MB LCM) → ~567 ms. ~10× payload ⇒ ~15× latency while the storage read stays ~5–8 ms. The direct-read numbers confirm it: ~1.4 % read, ~98.6 % base64 + JSON + HTTP.
Light endpoints stay cheap even on heavy ledgers. getTransaction / getEvents: ~1.8–2.9 ms on the pubnet tip → ~8–18 ms on 17 MB synthetic ledgers. They scale with the target ledger's size but never approach getLedgers' cost.
Cold ≈ hot — with one exception. The cold tier serves getLedgers, getTransaction, and dense-match getEvents at essentially hot-tier latency. The exception is selective multi-term getEvents on cold, which scales with term count (token ≈1 ms/term; soroswap ≈12 ms/term → 15 terms ≈ 137 ms) while hot stays flat — the POC's per-request cold-reader open + underfill re-query loop (fixable with an LRU + streaming pager).
Ingest gates freshness. 77–130 ms src→commit on synthetic profiles dwarfs the ~2–18 ms query legs, so how fast a brand-new ledger becomes queryable is ingest-bound for everything except getLedgers.

Where to spend on getLedgers (size-bound, not read-bound)

  1. Cap default/max page size ([serve].default_ledgers_limit / max_ledgers_limit).
  2. Transport compression (gzip/zstd Content-Encoding) — base64'd XDR is highly compressible.
  3. Stream the base64 encoder instead of materializing the full string per ledger.
  4. Lighter projection (header-only / opt-out of full meta).
  5. Storage-read tuning — the read is ~1 %; not worth it.
06

Caveats

Different hardware between parts
Part A ran on a 32 vCPU c6id.8xlarge / NVMe; Part B on a 16 vCPU Debian box / tmpfs. Compare shapes and ratios, not raw ms across parts.
Percentiles
Nearest-rank. Synthetic getLedgers p99 is over a budget-capped sample (71–127), less robust than the 500-rep getTransaction/getEvents p99s.
Cold reads are cache-warm
Synthetic cold reads are OS-page-cache-warm; an uncached first-touch adds disk latency.
Live-pubnet client contention
Client + server shared 16 cores (client stole CPU) and loopback RTT ≈ 0 — add real network latency to every p50 and expect higher ceilings on a dedicated client.
Concurrency
All Part A query passes are sequential (concurrency 1): unloaded-latency floors, not latency under concurrent load. Part B sweeps concurrency 1–64.
